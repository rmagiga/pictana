ここまで仕様が固まると、単なる「画像一覧高速化」ではなく、

```text
「状態遷移を含めた全体最適化」
```

として考える必要があります。

特にあなたのアプリは：

* Android + Windows
* SAF + USB OTG
* 10,000枚規模
* 検索/ソート/フィルター
* Grid + Viewer
* お気に入り
* スクロール復元
* プレロード

を持つので、

```text
「画面単位」
ではなく
「パイプライン単位」
```

で最適化するのが重要です。

---

# 最重要方針

あなたのケースでは：

```text
Flutter:
  UI orchestration

Native:
  image engine

SQLite:
  metadata index engine
```

の3層分離が最適です。

---

# 全体最適化アーキテクチャ

おすすめ構成です。

          │

# 1. 最大の改善ポイント

あなたの仕様では、

```text
画像decode
```

よりも実は：

```text
検索
ソート
フィルタ
一覧再構築
```

の方が危険です。

---

# NGパターン

Flutter側で：

```dart
images.where(...)
images.sort(...)
```

をやる。

---

# 理由

10,000件で：

* allocation
* rebuild
* GC
* isolate transfer

が発生。

---

# 正解

検索/ソート/フィルタは：

```text
Drift(SQLite)
```

へ完全委譲。

---

# 2. SQLiteを「検索エンジン化」

これがかなり重要。

---

# 推奨テーブル

```sql
images(
  entry_id INTEGER PRIMARY KEY,
  folder_id INTEGER,
  name TEXT,
  extension TEXT,
  modified INTEGER,
  size INTEGER,
  width INTEGER,
  height INTEGER,
  aspect_ratio REAL,
  thumb_key TEXT,
  indexed_at INTEGER
)
```

---

# index重要

```sql
CREATE INDEX idx_folder_name
ON images(folder_id, name);

CREATE INDEX idx_folder_modified
ON images(folder_id, modified);

CREATE INDEX idx_folder_ext
ON images(folder_id, extension);
```

---

# 効果

検索/ソートが：

```text
O(n log n)
```

↓

```text
SQLite index scan
```

になる。

かなり違う。
